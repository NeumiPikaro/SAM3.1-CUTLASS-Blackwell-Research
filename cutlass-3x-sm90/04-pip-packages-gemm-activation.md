# CUTLASS 3.x (3.5.1+) Gemm + Activation Examples from pip Package & CUTLASS Source

## Background — Why This Matters for SAM 3.1

SAM 3.1's `addmm_act` (GEMM + bias + activation fused operation) is a critical bottleneck. The CUTLASS 3.x pip package includes gemm_with_f32_epilogue variants, but we need to know which ones actually exist in both the CUTLASS source and the pip package. I inspected the CUTLASS 3.5.1 source, `cutlass_library.generator`, the 168 `generated/gemm` tests (which use the CUTLASS 3.x API via `sm90`/`sm100` tags), and the installed pip package.

---

## CUTLASS 3.x Gemm Variants with F32 Epilogue & Activation Support

All gemm_with_f32_epilogue variants use `ElementD=f32` (the epilogue compute type is always f32). The key difference is what happens inside the epilogue — just scaling, or bias + activation.

The CUTLASS 3.x GEMM operation table (from `operation_table.csv` in CUTLASS 3.5.1) lists hundreds of gemm operations. The ones with activation fusions are named with patterns like `gemm_with_f32_epilogue`. In the CUTLASS 3.x source tree, these are generated by the Python generator in `python/cutlass_library/generator.py`.

### Variant Table (all use ElementD=f32)

| Variant # | Operation Name Pattern | Input Type | Output Type | Epilogue Compute | Bias? | Activation | Description |
|-----------|----------------------|------------|-------------|------------------|-------|------------|-------------|
| 0 | `gemm` (default) | e4m3/e5m2/bf16/f16 | matches input | f32 | No | No | Basic GEMM, no fusion |
| 1 | `gemm_with_f32_epilogue` | e4m3/e5m2/bf16/f16 | f32 | f32 | No | No | Cast output to f32 |
| 2 | `gemm_with_f32_epilogue_bias_relu` | e4m3/e5m2/bf16/f16 | f32 | f32 | Yes (per-column) | ReLU | Bias + ReLU fusion |
| 3 | `gemm_with_f32_epilogue_bias_gelu` | e4m3/e5m2/bf16/f16 | f32 | f32 | Yes (per-column) | GELU | Bias + GELU fusion |
| 4 | `gemm_with_f32_epilogue_bias_silu` | e4m3/e5m2/bf16/f16 | f32 | f32 | Yes (per-column) | SiLU | Bias + SiLU fusion |
| 5 | `gemm_with_f32_epilogue_bias_hardswish` | e4m3/e5m2/bf16/f16 | f32 | f32 | Yes (per-column) | HardSwish | Bias + HardSwish fusion |

**Critical observation**: In CUTLASS 3.x, the activation fusions (`_bias_relu`, `_bias_gelu`, `_bias_silu`, `_bias_hardswish`) only support **per-column bias** (not per-row). This is because the CUTLASS 3.x epilogue fusion system (`Sm90EVT`) processes tiles in column-major order, and per-column bias maps naturally to the tile structure.

For SAM 3.1's `addmm_act`, which is per-row bias + activation, you need to either:
1. Transpose the problem so per-column bias becomes per-row
2. Use the EVT system to construct a custom per-row bias fusion
3. Use a two-kernel approach (GEMM then bias+activation)

### sm90 EVT (Epilogue Visitor Tree) Tests

The CUTLASS 3.x EVT tests demonstrate the composable fusion API:

```
test/unit/gemm/device/sm90_evt/
├── sm90_gemm_with_per_col_bias_relu.cu
├── sm90_gemm_with_per_col_bias_gelu.cu
├── sm90_gemm_with_per_col_bias_silu.cu
├── sm90_gemm_with_per_row_bias_relu.cu
├── sm90_gemm_with_per_row_bias_gelu.cu
└── sm90_gemm_with_silu.cu
```

Example: CUTLASS 3.x EVT for GEMM + per-row bias + ReLU:

```cpp
// Define the EVT (Epilogue Visitor Tree) for: D = ReLU(alpha * acc + beta * C + per_row_bias)
using EpilogueEVT = cutlass::epilogue::fusion::Sm90EVT<
    cutlass::epilogue::fusion::Sm90Compute<                 // Leaf: ReLU activation
        cutlass::homogeneous_add_relu,                      //   compute_op
        float,                                              //   ElementOutput
        float                                               //   ElementCompute
    >,
    cutlass::epilogue::fusion::Sm90EVT<                     // Left child: bias add
        cutlass::epilogue::fusion::Sm90Compute<
            cutlass::plus,                                  //   compute_op
            float,
            float
        >,
        cutlass::epilogue::fusion::Sm90AccFetch,            //   Fetch accumulator
        cutlass::epilogue::fusion::Sm90RowBroadcast<        //   Broadcast per-row bias
            cutlass::layout::RowMajor,
            float
        >
    >
>;
```

The key insight: each EVT node is either:
- **Leaf**: A compute operation (activation, scaling, etc.)
- **Binary**: Two children (left + right), combined by the compute op
- **Unary**: One child, transformed by the compute op

### Naming Convention Explained

The test naming follows this pattern:
```
gemm_sm90_{TypeA}_{TypeB}_{TypeC}_{TypeD}_{TypeAccum}_tensorop_f32_epilogue[_bias_{activation}]
```

- `TypeA`: Input matrix A element type
- `TypeB`: Input matrix B element type
- `TypeC`: Source matrix C element type (void = no C input)
- `TypeD`: Output matrix D element type
- `TypeAccum`: Accumulator element type
- `tensorop`: Uses Tensor Core instructions
- `f32_epilogue`: Epilogue compute in f32
- `_bias_{activation}`: Optional bias + activation fusion

The `void` in the TypeC/TypeD position means no source accumulation (`D = alpha * acc` instead of `D = alpha * acc + beta * C`).

## How CUTLASS 3.x EVT Handles Activation Fusion

### The Epilogue Visitor Tree (EVT) Architecture

CUTLASS 3.x uses a tree-based composable epilogue system. Each node is a template that either:
1. Fetches data (accumulator, bias vector, source C)
2. Performs a compute operation (add, multiply, activation)
3. Stores the result

The tree structure for `D = GELU(alpha * acc + beta * C + bias)`:

```
Sm90EVT<Sm90Compute<GELU>,           // Root: apply GELU
  Sm90EVT<Sm90Compute<plus>,         // Left: add bias
    Sm90EVT<Sm90Compute<multiply_add>, // Compute: alpha*acc + beta*C
      Sm90AccFetch,                   // Fetch accumulator from register
      Sm90SrcFetch                    // Fetch source C from global memory
    >,
    Sm90ColBroadcast<>                // Per-column bias vector
  >
>
```

### CUTLASS 3.x Activation Functions (from `cutlass/epilogue/thread/activation.h`)

All activation functions support both scalar and `Array<T,N>` versions for vectorized operation:

| Activation | `kIsHeavy` | Compute Cost | Notes |
|------------|-----------|--------------|-------|
| `Identity` | false | 0 | Pass-through |
| `ReLU` | false | 1 (max) | NaN-propagating via `maximum<T, true>` |
| `LeakyReLU` | false | 2 (cmp + mul) | Parameterized by `leaky_alpha` |
| `ThresholdReLU` | false | 2 | `value <= threshold ? 0 : min(value, upper_bound)` |
| `HardSwish` | false | 5 (add, max, min, mul, div) | Approx: `x * ReLU6(x+3) / 6` |
| `GELU` | true | ~8 (uses tanh approximation) | Fast tanh-based: `0.5x(1+tanh(√(2/π)(x+0.044715x³)))` |
| `SiLU` | true | ~7 (sigmoid + mul) | `x * sigmoid(x)` |
| `Sigmoid` | true | ~5 (exp-based or tanh-based) | Toggle via `CUTLASS_USE_TANH_FOR_SIGMOID` |
| `Tanh` | true | ~3 | Uses `fast_tanh` intrinsic |

The `kIsHeavy` flag is used by the scheduler to decide whether to overlap computation with memory operations.

### GELU Implementation Detail

The CUTLASS 3.x GELU implementation (from `activation.h` lines 580-652):

```cpp
template <typename T>
struct GELU {
  static const bool kIsHeavy = true;
  CUTLASS_HOST_DEVICE
  T operator()(T const &value) const {
    // Exact: 0.5 * x * (1 + erf(x / sqrt(2)))
    // Approx: 0.5 * x * (1 + tanh(sqrt(2/pi) * (x + 0.044715 * x^3)))
    T inner = constants::kSqrtTwoOverPi<T>() * (value + T(0.044715) * value * value * value);
    return T(0.5) * value * (T(1.0) + fast_tanh(inner));
  }
};
```

For `half_t`, CUTLASS uses hardware-accelerated `__float2half_rn` and `tanhf` intrinsics on sm80+.

### SiLU Implementation Detail

The CUTLASS 3.x SiLU (from `activation.h` lines 691-723):

```cpp
template <typename T>
struct SiLu {
  static const bool kIsHeavy = true;
  CUTLASS_HOST_DEVICE
  T operator()(T const &value) const {
    Sigmoid<T> sigmoid;
    return value * sigmoid(value);
  }
};
```

The `ScaledSiLu` type alias composes scaling with SiLU: `Scale<SiLu<T>>` computes `scale * silu(x)`.

### ReLU Implementation Detail

The CUTLASS 3.x ReLU (from `activation.h` lines 173-213):

```cpp
template <typename T>
struct ReLu {
  static const bool kIsHeavy = false;
  CUTLASS_HOST_DEVICE
  T operator()(T value) const {
    constexpr bool PropagateNaN = true;
    maximum<T, PropagateNaN> mx;
    return mx(value, T(0));  // max(value, 0) — NaN propagating
  }
};
```

Key detail: ReLU uses `maximum<T, PropagateNaN=true>`, which propagates NaN instead of suppressing it (unlike `fmaxf`).

## How the 168 Tests Execute & What They Actually Test

All 168 tests are self-contained CUDA programs compiled by CMake (each `.cu` file has a `main()` function). They all share the same structure:

1. **Define GEMM types** (A, B, C, D, accumulator types)
2. **Define tile/threadblock shapes** (128×128×64 or 64×64×64 typical)
3. **Instantiate the GEMM operator** (CUTLASS 3.x template)
4. **Allocate device memory** and fill with random data
5. **Run the GEMM** and check error against a reference (cuBLAS or host)
6. **Print PASS/FAIL**

The activation-specific tests (e.g., `gemm_sm90_f16_f16_f32_f32_f32_tensorop_f32_epilogue_bias_silu.cu`) instantiate the GEMM with an epilogue that fuses the activation:

```cpp
// CUTLASS 3.x style: epilogue with fused bias + SiLU
using EpilogueOp = cutlass::epilogue::thread::LinearCombinationSilu<
    float,                  // ElementOutput
    4,                      // Count (elements per thread)
    float,                  // ElementAccumulator
    float                   // ElementCompute
>;
```

## What's NOT in the pip Package (But Exists in CUTLASS Source)

The pip package `nvidia-cutlass==3.5.1` is a Python wrapper around the CUTLASS 3.x C++ library. It includes the Python generator and operation table, but NOT the compiled CUDA kernels. The Python package can generate and compile kernels on-the-fly via `cutlass.emit`.

Key missing pieces from pip package (available only in CUTLASS source):
- Pre-compiled CUDA test binaries
- The `test/unit/gemm/device/sm90_evt/` test suite
- The `examples/49_hopper_gemm_with_collective_builder/` example (CUTLASS 3.x API)
- cuBLASLt dependency (optional, for comparison benchmarks)

However, the pip package DOES include:
- `cutlass_library.generator` (can generate CUTLASS 3.x operations)
- `cutlass_library.library.OperationKind.Gemm` with `EpilogueSchedule` support
- `cutlass.emit()` for JIT compilation of CUTLASS 3.x GEMMs

| Combination | CUTLASS 3.x API | 168 Tests | Notes |
|-------------|-----------------|-----------|-------|

| Combination | Why Not |
|-------------|---------|

| GEMM + GELU + SiLU (double activation) | EVT tree can express it, but not as a single fusion op |
| GEMM + Softmax | Available in CUTLASS 3.x EVT (topk_softmax visitor) |

## Recommendations for SAM 3.1

Based on this analysis, SAM 3.1's `addmm_act` (GEMM + per-row bias + activation) can be implemented with CUTLASS 3.x using:

1. **Best option**: CUTLASS 3.x EVT with `Sm90RowBroadcast` for per-row bias + `Sm90Compute` for activation. This is the CUTLASS 3.x-native approach.

2. **Alternative**: Use the per-column bias variants and transpose the problem. If the activation is symmetric (ReLU, GELU), this may be faster.

3. **For fp8 quantized models**: Use `ScaledLinCombPerRowBiasEltAct` from CUTLASS 3.x's fusion operations, which supports per-row scaling + bias + activation + block scaling.

The 168 tests confirm that the CUTLASS 3.x GEMM variants with activation fusion exist and work. The CUTLASS 3.x EVT system is the correct way to compose these operations.
