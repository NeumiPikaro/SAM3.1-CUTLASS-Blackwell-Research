## 1. SM12.0 vs SM10.0 (Data Center Blackwell) — Architecture Overview

The Blackwell architecture spans two distinct compute capability levels that share the same microarchitectural DNA but differ in capability scope:

| Feature | SM10.0 (B200/GB200) | SM10.3 (B300/GB300) | SM12.0 (RTX 50 series) | SM12.1 (GB10/DGX Spark) |
|---|---|---|---|---|
| **Target** | Data center | Data center (refresh) | Consumer | Edge AI |
| **Process** | TSMC 4NP | TSMC 4NP | TSMC 4N | TSMC 4N |
| **Tensor Memory (TMEM)** | 256 KB per SM | 256 KB per SM | **Not available** | Not available |
| **Shared Memory (SMEM)** | Up to 228 KB | Up to 228 KB | Up to 100 KB | Up to 100 KB |
| **TMA (Tensor Memory Accelerator)** | Full TMA 2.0 | Full TMA 2.0 | **Yes, supported** | Yes, supported |
| **5th-gen Tensor Cores** | Yes | Yes | Yes | Yes |
| **FP4/FP6 Tensor Ops** | Yes | Yes | Limited | Limited |
| **NVLink** | NVLink 5.0 | NVLink 5.0 | No | No |
| **Distributed Shared Memory** | Yes (within cluster) | Yes | Yes (within cluster) | Yes |
| **GDDR Type** | HBM3e | HBM3e | GDDR7 | LPDDR5x |

### Critical Difference: No TMEM on SM12.0

The single most important architectural difference for kernel optimization is that **consumer Blackwell (SM12.0) does NOT include Tensor Memory (TMEM)**. TMEM is the dedicated 256 KB per-SM memory introduced in SM10.0 specifically for Tensor Core operand staging. On data center Blackwell, CUTLASS 3.x kernels use TMEM heavily to achieve peak tensor throughput.

On SM12.0, Tensor Cores operate on operands staged through **shared memory (SMEM)** and **register files**, similar to the Ada Lovelace (SM89) pattern. This fundamentally changes the kernel design space:
- No `tmem_alloc` / `tmem_dealloc` operations
- No TMEM-backed collective MMA pipelines
- Shared memory becomes the primary scratch space for both A/B operands AND partial results
- Tile scheduling strategies from SM10.0 (which assume TMEM) do not port directly

### Shared Memory Capacity

SM12.0 retains a per-SM shared memory pool configurable up to approximately **100 KB** (not the 228 KB available on SM10.0). The exact maximum depends on L1 cache / SMEM carveout configuration. This is a significant reduction:
- SM10.0: 228 KB SMEM (or shared with TMEM)
- SM12.0: ~100 KB SMEM max
- SM89 (Ada Lovelace): ~100 KB SMEM max
- SM90 (Hopper): ~228 KB SMEM max

For our SAM 3.1 kernel, this means tile sizes and double-buffering strategies must be designed around a ~100 KB budget, not the generous 228 KB of data center Blackwell.

### Key Takeaways for Kernel Optimization

1. **128-bit memory bus** — Memory bandwidth is the primary bottleneck. At ~448 GB/s, we're bandwidth-constrained on large matrix problems. Kernel efficiency in memory access patterns is paramount.

2. **32 MB L2 cache** — Generous for a mid-range GPU. This enables effective persistent kernel strategies and data reuse across thread blocks. The large L2 partially compensates for the narrower 128-bit bus.

3. **Low SM count (30-36)** — Limits total parallelism. We need high occupancy per SM to keep all SMs busy. Tile shapes that underutilize SMs will leave performance on the table.

4. **5th-gen Tensor Cores** — Available, supporting the same instruction set as SM10.0 (with TMEM-related instructions removed). See Section 3.

5. **High clock speeds (~2.5 GHz boost)** — Favorable for instruction throughput; FP32 compute potential is decent despite low SM count.

## 3. SM12.0 Tensor Core Capabilities

SM12.0 shares the 5th-generation Tensor Core ISA with SM10.0, but with the TMEM-related instructions removed. The supported operations include:

### Matrix Multiply-Accumulate (MMA) Instructions

| Instruction | Input Types | Accumulator | Shape | Notes |
|---|---|---|---|---|
| `mma.sync.aligned.m16n8k16` | FP16/BF16 × FP16/BF16 | FP16/BF16/FP32 | 16×8×16 | Standard half-precision |
| `mma.sync.aligned.m16n8k8` | TF32 × TF32 | FP32 | 16×8×8 | TF32 mode |
| `mma.sync.aligned.m16n8k32` | FP8E4M3 × FP8E4M3 | FP16/BF16/FP32 | 16×8×32 | FP8 mode |
| `mma.sync.aligned.m16n8k32` | FP8E5M2 × FP8E5M3 | FP16/BF16/FP32 | 16×8×32 | FP8 mode |
| `mma.sync.aligned.m16n8k64` | INT8 × INT8 | INT32 | 16×8×64 | Integer mode |
| `mma.sync.aligned.m16n8k256` | INT4 × INT4 | INT32 | 16×8×256 | Sub-byte integer |
| `mma.sync.aligned.m16n8k128` | E2M1 (FP4) × E2M1 | FP16/FP32 | 16×8×128 | 4-bit float |

### Block Scaling Support

SM12.0 supports **block-scaled (blockwise) MMA** operations, which is critical for quantized inference. The block scaling allows per-block dequantization factors to be applied during the MMA, avoiding separate scaling passes. This is the same capability as SM10.0.

### Warp-Level MMA vs MMA within Warpgroups

On SM10.0, CUTLASS uses **warpgroup-level MMA** (4 warps cooperating) backed by TMEM. On SM12.0:
- Standard warp-level MMA is the primary mode (`wgmma` instructions that used TMEM are NOT available)
- Cooperative MMA is achieved through shared memory staging + `ldmatrix` / `mma.sync` sequences
- The CUTLASS SM120 pingpong kernel (see Section 6) implements a software-based cooperative MMA pattern

### LdMatrix and StMatrix

SM12.0 retains `ldmatrix` (shared memory → register) and `stmatrix` (register → shared memory) instructions for efficient shared memory ↔ register transfers in the context of Tensor Core operand staging.

## 4. Shared Memory on SM12.0

### Capacity and Configuration

- **Maximum SMEM per SM:** ~100 KB (configurable carveout from L1)
- **Default SMEM allocation:** Usually 8-16 KB, configurable via `cudaFuncSetAttribute` with `cudaFuncAttributeMaxDynamicSharedMemorySize`
- **SMEM bank width:** 4 bytes (same as all recent architectures)
- **SMEM bank count:** 32 banks

### Comparison with SM10.0

| Property | SM10.0 | SM12.0 |
|---|---|---|
| Max SMEM | 228 KB | ~100 KB |
| TMEM | 256 KB | N/A |
| Usable for operands | SMEM + TMEM | SMEM only |
| Effective operand staging budget | 484 KB | ~100 KB |

The ~4.8× reduction in effective operand staging memory is the defining constraint of SM12.0 kernel design. This forces:
- **Smaller tile sizes** in the M and N dimensions
- **More conservative double-buffering** — fewer pipeline stages
- **Careful SMEM layout planning** to fit A + B + scaling factors + staging within 100 KB

### Practical SMEM Budget for a GEMM Kernel

For an FP16/BF16 GEMM with typical tile sizes on SM12.0:
- Assuming 100 KB total SMEM, and needing space for A and B tiles:
- If tile M=128, tile N=256, tile K=64: A tile = 128×64×2B = 16 KB, B tile = 64×256×2B = 32 KB
- Total for double-buffered A+B: (16+32)×2 = 96 KB — fits tightly
- Leaves minimal room for epilogue staging or additional pipeline buffers

This pushes us toward smaller tile configurations than what SM10.0 can achieve.

## 5. TMA (Tensor Memory Accelerator) Support on SM12.0

**Yes, SM12.0 supports TMA.** This is a critical feature that differentiates SM12.0 from SM89 (Ada Lovelace).

### What TMA Provides

The TMA unit is a dedicated hardware engine for asynchronous bulk memory transfers between global memory and shared memory. On SM12.0:
- **Bulk copy operations:** `cp.async.bulk` for efficient bulk transfers
- **TMA descriptors:** Support for creating TMA descriptors that describe tensor layouts, enabling hardware-managed address generation
- **Multicast support:** TMA can multicast to multiple SMs in a cluster
- **2D/3D/4D tensor copies:** Hardware can handle multi-dimensional strided copies without manual address calculation

### TMA on SM12.0 vs SM10.0

The SM12.0 TMA implementation is functionally equivalent to SM10.0's TMA 2.0 for consumer-class workloads. The main difference is:
- SM10.0 TMA can target both SMEM and TMEM; SM12.0 TMA targets SMEM only
- SM10.0 TMA has stronger integration with warpgroup MMA pipelines; SM12.0 requires manual staging

### CUTLASS TMA Support

CUTLASS 3.x's `CollectiveBuilder` supports SM12.0 TMA-backed copy operations. The `SM120_TMA_LOAD` and `SM120_TMA_STORE` operations are available. From the CUTLASS changelog:
- "Enable Blackwell SM120f compilation of examples" — confirming SM120 is a first-class target
- TMA descriptor management and TMA-based copy pipelines are functional on SM120

## 6. CUTLASS Examples Targeting SM12.0

### Known SM120-Specific Examples and Code

#### The SM120 Pingpong Kernel

The CUTLASS changelog explicitly mentions:

> "Fix missing SMEM alignment in Blackwell SM120 scale factors"

This confirms the existence of an **SM120-specific pingpong GEMM kernel** in CUTLASS. The pingpong schedule alternates data loading and computation between two pipeline stages, keeping the Tensor Core fed while the other stage loads the next data. On SM120 (without TMEM), this pattern is essential for overlapping memory access with compute.

The SM120 pingpong kernel likely lives in the CUTLASS include path under:

## 7. Known Limitations of SM12.0 vs SM10.0

### Summary of Limitations

| Capability | SM10.0 | SM12.0 | Impact |
|---|---|---|---|
| **Tensor Memory (TMEM)** | 256 KB | ❌ Not available | Cannot use warpgroup MMA pipeline; must use warp-level MMA + SMEM staging |
| **SMEM capacity** | 228 KB | ~100 KB | Smaller tiles, fewer pipeline stages |
| **Warpgroup MMA** | `wgmma` instructions | ❌ Not available | Must use `mma.sync` with `ldmatrix` pipeline |
| **NVLink** | NVLink 5.0 | ❌ Not available | No multi-GPU coherent shared memory |
| **Distributed SMEM scope** | Cross-SM clusters | Cross-SM clusters (limited) | Cluster size may differ |
| **HBM bandwidth** | ~8 TB/s (HBM3e) | ~448 GB/s (GDDR7) | 18× bandwidth gap — bandwidth is the dominant constraint |
| **SM count** | 132-192 SMs | 30-36 SMs | Much less total parallelism |
| **FP4/FP6 precision** | Full support | Partial / limited | Some exotic quantization modes may not work |
| **Async pipeline depth** | Deep (TMEM-backed) | Shallow (SMEM-limited) | Less overlap between compute and memory |

### What Works Well on SM12.0 Despite Limitations

1. **TMA is available** — Asynchronous bulk transfers are fully functional. This is the single biggest advantage over SM89.
2. **Block-scaled MMA** — Per-block dequantization in hardware is supported, enabling efficient quantized inference.
3. **FP8 native support** — E4M3 and E5M2 formats are first-class Tensor Core inputs.
4. **32 MB L2 cache** — Generous for data reuse; persistent kernel strategies can leverage this.
5. **High clock speeds** — ~2.5 GHz boost compensates partially for lower SM count.
6. **GDDR7 memory** — First consumer GPU with GDDR7; modest bandwidth improvement over GDDR6X.
7. **5th-gen Tensor Core ISA** — Same fundamental instruction set as SM10.0 (minus TMEM ops).

## 8. Implications for SAM 3.1 Kernel Configuration

### Recommended Kernel Architecture

Given the constraints of SM12.0 (no TMEM, ~100 KB SMEM, 30-36 SMs, 128-bit memory bus), the SAM 3.1 kernel should be designed as follows:

#### Tile Size Selection

For FP16/BF16 GEMM on SM12.0:
- **Recommended CTA tile:** M=128, N=128, K=32 (or M=128, N=64, K=64)
- **Warp tile:** M=64, N=64, K=32
- **Rationale:** Must fit double-buffered A+B within ~100 KB. M=128, N=128, K=32 with FP16: A=128×32×2B=8KB, B=32×128×2B=8KB, double-buffered=32KB. Leaves ~68KB for epilogue staging, barriers, and overhead.

For FP8 GEMM:
- **Recommended CTA tile:** M=128, N=256, K=64
- FP8 halves operand storage, allowing larger tiles
- A=128×64×1B=8KB, B=64×256×1B=16KB, double-buffered=48KB

#### Pipeline Strategy

Since SM12.0 lacks TMEM, use the **software pipelining** pattern:
1. **TMA-based prefetching** into SMEM (leverage SM12.0's TMA support)
2. **`ldmatrix` from SMEM to registers** for Tensor Core operand staging
3. **`mma.sync` accumulation** in register files
4. **Double-buffered SMEM** for overlapping load and compute

The CUTLASS SM120 pingpong kernel is the reference pattern — our kernel should follow its scheduling model.

#### Collective Mainloop Configuration

In CUTLASS 3.x terms, configure:
```cpp
// SM120 Collective Mainloop (Pingpong schedule)
using CollectiveMainloop = cutlass::gemm::collective::CollectiveMma<
    cutlass::gemm::MainloopSm120TmaGmmaPingpong<
        ElementA, LayoutA, AlignmentA,
        ElementB, LayoutB, AlignmentB,
        ShapeTileMNK,   // e.g., Shape<_128, _128, _32>
        StageCount,      // 2 (double-buffered)
        Scheduler        // clc or default
    >
>;
```

#### Epilogue Considerations

- Use vectorized epilogue writes (128-bit stores) to maximize memory bandwidth utilization
- The 128-bit bus means epilogue bandwidth is ~448 GB/s — every byte counts
- Fused epilogue operations (bias, activation) are essential to avoid extra memory passes

#### Cluster Configuration

- SM12.0 supports thread block clusters but the effective distributed SMEM is limited
- Recommend cluster size of 2-4 for most kernels
- Larger clusters may exceed SMEM budget or cause scheduling inefficiency on 30-36 SM systems

#### Occupancy Optimization

- With only 30-36 SMs, we need high occupancy per SM to saturate the GPU
- Target ≥50% occupancy (≥24 warps per SM with 128 max)
- Avoid register-heavy configurations that reduce occupancy
- Monitor register pressure: 255 register limit per thread; aim for ≤128 registers per thread for ≥2 blocks/SM

### Specific CUTLASS Compilation Flags

When compiling CUTLASS for SM12.0:
- Use `-gencode arch=compute_90a,code=sm_120a` (or `sm_120f`)
- The `a` suffix enables architecture-specific optimizations
- SM120f is the "full" variant (all consumer features)
- CUDA Toolkit 12.8+ required for SM12.0 compilation

### FP8 Optimization Path

For SAM 3.1, FP8 (E4M3) is the ideal precision path on SM12.0:
- Halves SMEM operand requirements → larger tiles possible
- Doubles effective Tensor Core throughput (32 K-dimension MMA)
- Block-scaled dequantization in hardware avoids separate scaling kernels
- NVFP4 support opens further quantization if precision permits

|---|---|---|
| Tensor Core generation | 4th (Ada) | 5th (Blackwell) |
